# Sphinx Documentation Migration — Design

**Status:** Draft for review
**Date:** 2026-05-22
**Owner:** Davor Runje
**Triggering event:** MkDocs build warning announcing that MkDocs is unmaintained and
that an incompatible "MkDocs 2.0" is planned; the `mkdocs-material` maintainer is also
exiting toward new projects (MaterialX / Zensical).

## Goal

Future-proof the `mononet` documentation pipeline by migrating from MkDocs +
`mkdocs-material` to Sphinx + PyData Sphinx Theme. End the dependency on the
fragmenting MkDocs ecosystem and align the docs stack with the Python ML ecosystem
that `mononet` targets (PyTorch, JAX, Keras — all of which use Sphinx).

## Non-goals

- Content rewrites beyond syntax conversion (no rewording of guides, concepts, etc.).
- Visual redesign beyond what the PyData theme provides out of the box.
- Evaluation or adoption of Zensical / MaterialX (revisit in 12+ months only if
  Sphinx proves unsuitable — Sphinx makes this moot in practice).
- Switching docs hosting away from GitHub Pages.
- Adding new documentation pages or sections.

## Background

The current docs stack (as of 2026-05-22):

- Engine: `mkdocs` (pulled in transitively via `mkdocs-material`, not directly
  pinned), theme: `mkdocs-material==9.7.6`.
- Seven plugins: `mkdocstrings` (API docs), `mkdocs-literate-nav` (SUMMARY.md nav),
  `mkdocs-glightbox` (image zoom), `mkdocs-jupyter` (notebooks),
  `mkdocs-git-revision-date-localized-plugin`, `mkdocs-minify-plugin`, `mike`
  (versioning).
- Custom orchestration in `docs/docs.py` (Typer CLI) that imports
  `mkdocs.commands.build`, `mkdocs.commands.serve`, and `mkdocs.config` directly,
  and shells out to `mkdocs build`.
- Custom hook `docs/create_api_docs.py` that generates API reference pages and
  rebuilds `SUMMARY.md` based on the package layout.
- Content: 13 hand-authored Markdown pages (`guides/`, `concepts/`, `about/`,
  `index.md`, `contributing.md`), 1 Jupyter notebook (`benchmarks/00-overview.ipynb`),
  ~16 auto-generated API reference pages.
- Published to GitHub Pages at `https://davorrunje.github.io/mononet/`.

Decisions taken during brainstorming (in order):

1. **Goal:** Future-proof the whole docs pipeline (not just silence the warning).
2. **Disruption tolerance:** Anything goes — pick the best long-term answer.
3. **Engine:** Sphinx (over ProperDocs continuation, Zensical, or another SSG).
4. **Theme:** PyData Sphinx Theme (over Furo or Sphinx Book Theme).
5. **API generator:** `sphinx-autodoc2` (over `sphinx-autoapi` or classic
   `autodoc + autosummary`).
6. **Versioning:** `sphinx-multiversion` (over Read the Docs or dropping versioning).
7. **Migration strategy:** In-place replacement (over parallel build or worktree).

## Architecture

├── index.md                   # Landing page (MyST)
├── guides/
│   ├── pytorch.md
│   ├── jax.md
│   └── keras.md
├── concepts/
│   ├── monotonicity.md
│   └── layers.md
├── benchmarks/
│   ├── index.md

├── about/
│   ├── license.md
│   ├── changelog.md
│   └── citation.md
├── contributing.md
├── apidocs/                   # GENERATED by sphinx-autodoc2 (gitignored)
├── _static/                   # CSS, logo, favicon (replaces overrides/)

### Removed during migration

- `docs/mkdocs.yml`
- `docs/docs.py`
- `docs/create_api_docs.py`
- `docs/docs/SUMMARY.md` (replaced by Sphinx toctree directives in `index.md`)
- `docs/overrides/` (replaced by `docs/_templates/` + `docs/_static/`)
- `docs/site/` (build output, never committed but tracked in `.gitignore`)
- `docs/__init__.py` (the `docs` directory is no longer a Python package)
- `docs/docs/` nested directory — content moves up one level into `docs/`.

the `create_api_docs.py` hook. Net: less custom code, fewer moving parts.

## Components

### Authoring layer (MyST Markdown)

Existing `.md` files require mechanical syntax conversion:

| MkDocs / pymdownx                  | Sphinx / MyST                                  |
|------------------------------------|------------------------------------------------|
| `!!! note "Title"`                 | `` ```{note}`` block (or `:::{note}` fence)     |
| `=== "PyTorch"` (`pymdownx.tabbed`) | `` ```{tab-set} `` / `` ```{tab-item} `` via `sphinx-design` |
| `# (1)` code annotations           | Inline comments or `sphinx-design` callouts (small loss) |
| `mdx_include` file inclusion       | MyST `{include}` directive                     |
| `pymdownx.emoji` (Material icons)  | `sphinx-design`'s icon system or plain Unicode |
| `pymdownx.details`                 | `sphinx-togglebutton` + admonitions            |
| `pymdownx.highlight` anchor lines  | Built into Sphinx code-block directive         |

**Capability gains:**

- Inline math via MyST `dollarmath` extension (`$x^2$`) — not currently available.
- `intersphinx` cross-links into PyTorch/JAX/Keras/NumPy docs from docstrings and
  prose — not currently available.

### API documentation (`sphinx-autodoc2`)

- Configured to point at `../mononet` and traverse the package automatically.
- Generates one MyST page per module/class/function under `docs/apidocs/`.
- Output is gitignored; regenerated on every build.
- Google-style docstrings preserved (matches existing project style per
  [CLAUDE.md](../../../CLAUDE.md)).
- Output respects the strict mypy types already present in the package — type
  annotations become clickable cross-references via `intersphinx`.

The hand-authored navigation under "Reference - Code API" in the current
`SUMMARY.md` is removed. The new API navigation is generated from the package
structure by `autodoc2` and referenced from a top-level `toctree` in `index.md`.

### Notebook integration (`myst-nb`)

- `benchmarks/00-overview.ipynb` is rendered directly into the site.
- `nb_execution_mode = "off"` matches the current `execute: false` setting:
  notebooks render with their committed outputs and are not re-executed at build
  time.
- Future option: switching to `nb_execution_mode = "cache"` would cache outputs
  by cell hash — strictly better than the current all-or-nothing model — but is
  out of scope for this migration.

### Versioning (`sphinx-multiversion`)

- Builds docs for each git branch/tag matching the whitelist patterns. The
  existing `mike`-based aliasing scheme is preserved in spirit:
  - `main` branch → `dev` (aliased as `latest`).
  - tags matching `v*.*.*` → `<version>` (latest tag aliased as `stable` and set
    as default).
- Output structure: `_build/html/dev/`, `_build/html/v0.1.0/`, …, plus alias
  symlinks/copies for `latest` and `stable`.
- A `versions.json` file is generated post-build and consumed by the PyData
  theme's version-switcher dropdown. `sphinx-multiversion` does not emit this
  file natively; a small post-build step writes it from the discovered build
  directories.
- The deploy step pushes the entire `_build/html/` tree to the `gh-pages` branch.
- Since the library has no tagged releases yet, the initial publish contains
  only `dev`/`latest`. The version switcher becomes meaningful once the first
  tag exists.

### Build orchestration

- `tools/build-docs.sh` becomes a thin wrapper:
  ```bash
  #!/usr/bin/env bash
  set -e
  set -x
  cd docs
  uv run sphinx-multiversion . _build/html
  ```
  For single-version dev builds: `uv run sphinx-build . _build/html`.
- `tools/serve-docs.sh` becomes:
  ```bash
  #!/usr/bin/env bash
  set -e
  set -x
  cd docs
  uv run sphinx-autobuild . _build/html --port 8008 --host 0.0.0.0
  ```
- No Python orchestration layer (`docs.py`, `create_api_docs.py`) remains.

### Dependencies (`pyproject.toml`)

Remove from the `docs` dependency group:
- `mkdocs-material==9.7.6`
- `mkdocstrings==1.0.4`
- `mkdocstrings-python==2.0.3`
- `mkdocs-literate-nav==0.6.3`
- `mkdocs-glightbox==0.5.2`
- `mkdocs-jupyter>=0.25`
- `mkdocs-git-revision-date-localized-plugin==1.5.2`
- `mkdocs-minify-plugin==0.8.0`
- Any other `mkdocs-*` or `mkdocs` deps.

Add:
- `sphinx`
- `pydata-sphinx-theme`
- `sphinx-autodoc2`
- `myst-nb` (pulls in `myst-parser` and Jupyter machinery)
- `sphinx-copybutton`
- `sphinx-design`
- `sphinx-togglebutton`
- `sphinx-multiversion`
- `linkify-it-py` (required by MyST's `linkify` extension)

Version pins to be set during implementation based on current PyPI releases at
that time.

### CI / deploy

- The GitHub Actions workflow at `.github/workflows/docs.yml` (currently uses
  `mike deploy --push --update-aliases ...` against the `gh-pages` branch) is
  updated to:
  - Install the new docs dependency group.
  - Run `sphinx-multiversion docs docs/_build/html`.
  - Generate `versions.json` from the build output.
  - On `main`: alias the `main` build directory as `dev` and `latest`, then push
    to `gh-pages`.
  - On a `v*.*.*` tag: alias that tag's build directory as the version number,
    set the most recent tag as `stable`, mark it as default, then push to
    `gh-pages`.
  - The mechanism (raw `git push` of a built tree, `peaceiris/actions-gh-pages`,
    or `actions/deploy-pages`) is decided during implementation. The existing
    workflow uses a direct `git push` via `mike`.
- The existing `git-revision-date-localized` "last updated" stamp on each page is
  replaced by Sphinx's built-in `html_last_updated_fmt` (uses git or filesystem
  mtime). Not a perfect match (no "X days ago" relative time) but acceptable for
  parity.

## Data flow

```
source ─┬─ docs/*.md, docs/*.ipynb ─────────────┐
        │                                       │
        └─ mononet/ (Python package) ──► sphinx-autodoc2 ──► docs/apidocs/*.md
                                                            │
                                                            ▼
                                          MyST parser (myst-nb)
                                                            │
                                                            ▼
                                              Sphinx build pipeline
                                                            │
                                                            ▼
                                          docs/_build/html/  (per-version)
                                                            │
                                                            ▼
                                              gh-pages branch  ──► GitHub Pages
```

## Error handling & build hygiene

- Builds run with `-W` (warnings-as-errors) once stabilized. During the initial
  migration the flag may be temporarily off to triage cumulative warnings, but
  the merge gate requires `-W` clean.
- Broken cross-references (intersphinx misses, missing toctree entries) become
  build failures, not silent gaps.
- The deploy step refuses to publish if the build emitted any warnings under
  `-W`.

## Testing & verification

A migration passes when **all** of the following hold:

1. **Build correctness**
   - `uv run sphinx-build -W docs docs/_build/html` exits 0 with no warnings.
   - `uv run sphinx-multiversion docs docs/_build/html` succeeds for `main`.
2. **Content parity**
   - Every page in the current `SUMMARY.md` (13 hand-authored pages + 1 notebook
     + ~16 API ref pages) is reachable in the new site via the navigation.
   - Spot-check at least one page per top-level section renders correctly:
     `index.md`, one `guides/`, one `concepts/`, `benchmarks/index.md`,
     `benchmarks/00-overview.ipynb`, one API page, one `about/`,
     `contributing.md`.
   - API reference includes every public symbol in `mononet`, `mononet.core`,
     `mononet.jax`, `mononet.torch` — counted and compared against the current
     `SUMMARY.md` API section.
3. **Feature parity**
   - Code blocks have a copy button.
   - Light/dark mode toggle works.
   - Tabs (per-framework guide examples) render correctly.
   - Notebook outputs render.
   - Edit-on-GitHub link present on every page.
   - Search works (built-in Sphinx search; no extra config).
4. **New capabilities verified**
   - Add a smoke-test page (later removed) containing `$\mathbb{R}^n$` and
     `\begin{align} y = f(x) \end{align}` — confirm rendering.
   - In one API page, confirm a type annotation like `torch.Tensor` is a
     clickable cross-reference to the PyTorch docs.
5. **Tooling**
   - `./tools/build-docs.sh` succeeds.
   - `./tools/serve-docs.sh` serves on `http://0.0.0.0:8008` with live reload.
   - `./tools/build-docs.sh` no longer emits the MkDocs warning.
6. **CI**
   - The docs workflow runs to green on a PR branch.
   - Generated `gh-pages` content is browsable on a preview deployment (or, if
     unavailable, locally via `python -m http.server` on the build output).
7. **No regressions**
   - `uv run pytest` still passes (docs migration should not touch package code).
   - `uv run mypy` still passes.
   - Pre-commit hooks pass.

## Open questions

None blocking. The package-name / module-import behavior of any MkDocs-family
fallback was researched during brainstorming but is moot — we are leaving the
ecosystem entirely, not switching to ProperDocs.
