# Deploy Documentation to GitHub Pages — Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Get `mononet` documentation actually serving at `https://davorrunje.github.io/mononet/` with a working PyData version switcher, by (a) replacing the unmaintained `sphinx-multiversion` with `sphinx-polyversion`, and (b) replacing the `gh-pages` branch deploy with GitHub's official Pages Actions pipeline.

**Architecture:** Five code/config changes then one CI roll. Tasks 1–4 fix the build locally (swap dep, add driver, drop dead config, validate). Task 5 rewrites the workflow to publish via `actions/configure-pages` → `actions/upload-pages-artifact` → `actions/deploy-pages`. Tasks 6–7 push + verify the live site. Task 8 cleans up the legacy `gh-pages` branch.

**Tech Stack:** Sphinx, `sphinx-polyversion`, PyData Sphinx Theme, `uv`, GitHub Actions (`actions/configure-pages@v5`, `actions/upload-pages-artifact@v3`, `actions/deploy-pages@v4`).

**Spec:** [docs/superpowers/specs/2026-05-22-deploy-docs-gh-pages-design.md](../specs/2026-05-22-deploy-docs-gh-pages-design.md)

## Task 2: Create the `sphinx-polyversion` driver `docs/poly.py`

**Files:**
- Create: `docs/poly.py`

This driver tells `sphinx-polyversion` which branches/tags to build, where to write output, and how to invoke `sphinx-build`. It replaces what `smv_*` keys used to do in `conf.py`.

- [ ] **Step 1: Inspect the installed `sphinx-polyversion` API to confirm class names**

Before writing the driver, confirm the exact import surface of the installed version (the API has stabilised but it's worth verifying once):

```bash
uv run python -c "import sphinx_polyversion, sphinx_polyversion.git, sphinx_polyversion.pyvenv, sphinx_polyversion.sphinx; print('OK')"
uv run python -c "from sphinx_polyversion import DefaultDriver; from sphinx_polyversion.git import Git, file_predicate; from sphinx_polyversion.pyvenv import Pip; from sphinx_polyversion.sphinx import SphinxBuilder; print('imports OK')"
```

Expected: both print `OK` / `imports OK`. If any import fails, find its replacement by reading `uv run python -c "import sphinx_polyversion; print(sphinx_polyversion.__file__)"` and inspecting the installed package directory. Adjust the imports in Step 2 to match.

- [ ] **Step 2: Write `docs/poly.py`**

Create `docs/poly.py` with the following content:

```python
"""sphinx-polyversion driver for mononet docs.

Builds the ``main`` branch plus any ``v<major>.<minor>.<patch>`` tag into
``docs/_build/html/<ref-name>/``. Each per-ref build receives ``DOCS_VERSION``
in its environment so ``docs/conf.py`` can set the PyData switcher's
``version_match`` correctly.
"""

from __future__ import annotations

from pathlib import Path

from sphinx_polyversion import DefaultDriver
from sphinx_polyversion.git import Git, file_predicate
from sphinx_polyversion.pyvenv import Pip
from sphinx_polyversion.sphinx import SphinxBuilder

#: Regex matching branches to build.
BRANCH_REGEX = r"^main$"

#: Regex matching tags to build (semver tags only).
TAG_REGEX = r"^v\d+\.\d+\.\d+$"

#: Only consider refs from this remote.
REMOTE_REGEX = r"^origin$"

#: Build output dir, relative to repo root.
OUTPUT_DIR = "docs/_build/html"

#: Sphinx source dir, relative to repo root.
SOURCE_DIR = "docs"

#: Arguments forwarded to ``sphinx-build``.
SPHINX_ARGS = ["-W"]

src = Path(SOURCE_DIR)
root = Path(__file__).resolve().parent.parent

DefaultDriver(
    root=root,
    out=OUTPUT_DIR,
    vcs=Git(
        branch_regex=BRANCH_REGEX,
        tag_regex=TAG_REGEX,
        remote_regex=REMOTE_REGEX,
        buffer_size=1 * 10**9,  # 1 GB; large enough for the worktree exports
        predicate=file_predicate([src]),
    ),
    builder=SphinxBuilder(src, args=SPHINX_ARGS),
    env=Pip.factory(args=[".", "--group", "docs"]),
).run()
```

Notes for the implementer:
- `Pip.factory(args=[".", "--group", "docs"])` tells polyversion to create a venv per ref and `pip install . --group docs` inside it. This requires `pip >= 25.1` (PEP 735 support). If the polyversion-created venv ships an older pip, the install will fail with "unrecognized arguments: --group"; in that case, change `args` to `[".[all]"]` and add a `docs` extra in `pyproject.toml` that mirrors the `docs` group. Confirm this in Task 4 Step 2.
- The `predicate=file_predicate([src])` filter skips refs that don't contain a `docs/` directory. This protects against pre-Sphinx-migration tags being attempted.

- [ ] **Step 3: Commit**

```bash

git commit -m "docs(build): add sphinx-polyversion driver"
```

## Task 5: Rewrite `.github/workflows/docs.yml` to use the official Pages Actions

**Files:**
- Modify: `.github/workflows/docs.yml` (full rewrite)

- [ ] **Step 1: Replace the workflow file**

Overwrite `.github/workflows/docs.yml` with:

```yaml
name: Docs

on:
  push:
    branches: [main]
    tags: ['v*.*.*']
  workflow_dispatch:

concurrency:
  group: docs-${{ github.ref }}
  cancel-in-progress: true

jobs:
  build:
    runs-on: ubuntu-latest
    permissions:
      contents: read
    steps:
      - uses: actions/checkout@v6
        with:
          fetch-depth: 0
          fetch-tags: true

      - uses: actions/setup-python@v6
        with:
          python-version: "3.13"

      - uses: astral-sh/setup-uv@v7

      - name: Install dependencies
        run: uv sync --group docs --extra all

      - name: Build all versions
        run: |
          git config user.name  "github-actions[bot]"
          git config user.email "github-actions[bot]@users.noreply.github.com"
          uv run sphinx-polyversion docs/poly.py

      - name: Generate versions.json
        run: |
          uv run python tools/gen_versions_json.py \
            docs/_build/html https://davorrunje.github.io/mononet

      - name: Write root index.html redirecting to /main/
        run: |
          cat > docs/_build/html/index.html <<'EOF'
          <!doctype html>
          <meta charset="utf-8">
          <meta http-equiv="refresh" content="0; url=./main/">
          <link rel="canonical" href="./main/">
          EOF

      - name: Configure Pages
        uses: actions/configure-pages@v5
        with:
          enablement: true

      - name: Upload Pages artifact
        uses: actions/upload-pages-artifact@v3
        with:
          path: docs/_build/html

  deploy:
    needs: build
    runs-on: ubuntu-latest
    permissions:
      pages: write
      id-token: write
    environment:
      name: github-pages
      url: ${{ steps.deployment.outputs.page_url }}
    steps:
      - name: Deploy to GitHub Pages
        id: deployment
        uses: actions/deploy-pages@v4
```

Key differences from the existing workflow:
- Two jobs (`build` + `deploy`) instead of one — required by the Pages Actions model.
- Permissions split: `build` is `contents: read`; `deploy` holds `pages: write` + `id-token: write`.
- No `peaceiris/actions-gh-pages`; no `gh-pages` branch is written.
- `actions/configure-pages@v5` with `enablement: true` will create the Pages site on the first run if it doesn't exist yet.
- `actions/deploy-pages@v4` declares the `github-pages` deployment environment, which surfaces the deploy URL in the run summary.
